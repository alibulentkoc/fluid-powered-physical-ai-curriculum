## 2. The concept: documentation and reflection

**The documentation deliverables.** A complete machine is captured in a standard set of documents:

- **Final hydraulic schematic** — the complete ISO 1219 circuit diagram with a component list. The definitive description of the machine's hydraulic design (the standard from Module 02 onward).
- **Wiring diagram** — every sensor, controller, and power connection. How the machine's nervous system is wired.
- **Code repository** — all Arduino firmware and Python scripts, documented and version-controlled. The machine's intelligence, reproducible.
- **Digital twin** — the full twin with its validation against demonstration data. The machine's digital counterpart, ready to monitor and extend.
- **Validation report** — measured performance vs. specifications, residual analysis, fault-detection results. The proof the machine works (Lesson 03), recorded.
- **Final report** — system description, design decisions, results, critical reflection, future directions. The comprehensive account of the machine.
- **Demonstration video** — the machine performing its tasks, the twin displayed. The machine shown in action.

Together these make the machine *transferable*: another engineer can understand it, reproduce it, maintain it, and extend it. This is what turns a working machine into an engineering deliverable.

**Critical reflection.** Beyond documentation, the capstone asks the engineer to reflect:
- **What worked, and why?** The design decisions that paid off.
- **What would be done differently?** The honest assessment of weaknesses and what a redesign would change — the hallmark of an engineer who understands their work, not just completed it.
- **What does the platform enable?** At least one research or extension direction the machine now makes possible — because a good platform is a foundation for further work, not an endpoint.

This reflection demonstrates *understanding* — the ability to describe, defend, and extend the machine, which is the true measure of an engineer's grasp of their work.

**The platform as a foundation.** The Smart Agricultural Workcell is not just a finished project; it is a *platform* — a working intelligent fluid-powered machine that enables further work: camera-guided manipulation, learned control, multi-actuator coordination, field deployment, new agricultural tasks. The reflection identifies where it can go next, framing the capstone not as an end but as a beginning.

## 3. Mathematical model

**Documentation completeness as coverage.** A complete deliverable covers every subsystem and every claim:
$$\text{complete} = \bigwedge_{\text{subsystem}} \text{documented} \;\wedge\; \bigwedge_{\text{spec}} \text{validated and reported}$$
Every subsystem has a schematic/diagram, every specification has a measured result in the validation report — no gaps.

**Reproducibility.** The deliverable is reproducible when another engineer, given only the documentation, can rebuild and re-validate the machine:
$$\text{reproducible} \iff \text{(schematic + wiring + code + parameters + procedure)} \to \text{rebuilt machine meets same specs}$$
This is the test of good documentation: it contains everything needed to reproduce the result.

**The validation report's core table.** The heart of the validation report maps each specification to its measured result and verdict — the seven-row table from Lesson 03, the machine's proof of performance, recorded permanently.

## 11. Common mistakes

**Treating documentation as an afterthought.** Documentation is half the deliverable; a working machine without it is a one-off that cannot be transferred, maintained, or extended. Document as you build, finalize at the end.

**Reporting only successes.** Honest engineering documents weaknesses and missed specs with plans to address them. A report that hides shortfalls is not trustworthy and not useful.

**Confusing "it works" with "I understand it."** Passing specs is not the same as understanding the machine. Reflection — why decisions were made, where the weaknesses are, what's next — demonstrates the understanding that completes the engineer's work.

**Treating the capstone as an endpoint.** The workcell is a platform for further work. A good capstone identifies where it can go next, framing the project as a foundation, not a finish.
